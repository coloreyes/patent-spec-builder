# 专利说明书生成（主流程）

## 用途

将技术交底书或技术论文转换为标准**专利说明书**格式，输出两份文件：
1. **说明书摘要** — 一段式，200-400字
2. **说明书** — 完整格式，包含技术领域、背景技术、发明内容、附图说明、具体实施方式

## 输出格式要求

### 文件一：说明书摘要

**格式**：单一段落，不分节标题，约200-400字。

### 文件二：说明书

**章节结构**（严格按以下顺序，无编号前缀）：

```
[发明名称]（首行，居中，黑体16pt加粗）

技术领域（黑体14pt加粗）
背景技术（黑体14pt加粗）
发明内容（黑体14pt加粗）
附图说明（黑体14pt加粗）
具体实施方式（黑体14pt加粗）
```

### 引用与术语规范（核心）

**专利说明书不使用 `[1]`, `[2]` 等参考文献编号，不使用独立的参考文献章节。**

所有外部内容（术语、数据集、基线方法、学术概念）必须通过**内联解释**紧跟在所解释的内容之后：

1. **英文术语**：中文名称后紧跟 `（English Term）`，如：
   - `多模态特征融合（Multimodal Feature Fusion）：指...`
   - `概念漂移（Concept Drift）：指...`

2. **术语解释格式**：在「具体实施方式」开头部分，每个术语独立成段：
   ```
   术语中文名（English Term）：通用定义。[可选：在本发明中，...]
   ```
   - 术语数量根据实际正文内容确定，提取所有必要的专业术语
   - 部分术语需要追加「在本发明中，[该术语如何被使用]」

3. **数据集描述格式**：数据集名称后用括号描述其内容：
   ```
   HateMM（包含1083个来自 BitChute 和 Odysee 的视频，涵盖英文多模态仇恨文本、视觉和音频信息）
   MultiHateClip-YouTube（简称 MHClip-A，涵盖 YouTube 平台上的多语言短视频）
   ```

4. **基线方法描述格式**：每个基线方法独立成段，直接解释其原理：
   ```
   ER：传统的经验回放机制，通过简单存储先前的少量样本进行重新训练以维持旧知识...
   LwF：基于知识蒸馏的非重放方法，依靠蒸馏前一阶段旧模型对于旧特征输出映射来减缓覆写遗忘。
   ```

### 撰写风格规范

1. **不使用**「一、二、三」或「第一章」等数字编号作为节标题
2. **节标题**仅用：`技术领域`、`背景技术`、`发明内容`、`附图说明`、`具体实施方式`
3. **步骤编号**使用 `S1`、`S2`...`Sn` 格式
4. **有益效果编号**使用 `（1）`、`（2）`... 格式
5. **公式**使用 Word 原生 OMath 格式
6. **表格**使用 Word 原生表格，标题用 Caption 样式
7. **术语**首次出现时给出中英文全称，如「持续学习（Continual Learning, CL）」
8. **段落**首行缩进0.74cm，行距1.5倍，宋体12pt
9. **实施例1 展开**：每个步骤（S1-Sn）在实施例1中需展开为 3-5 段详细描述
10. **结语**以「本领域的普通技术人员将会意识到...」收尾

### Word 生成技术要求

- 使用 `python-docx` 生成
- 公式使用 **OMath XML 注入**：使用 `tools/omath_converter.py` 中的 `formula_to_omath(latex)` 函数
- 正文段落使用 `style='Normal'`，首行缩进 `0.74cm`，行距 1.5 倍
- 节标题使用黑体 14pt 加粗，正文使用宋体 12pt
- 参考文献段落使用宋体 10.5pt，首行不缩进

## 转换流程

1. **Read** 交底书全文及原始论文（如可用）
2. **提取**所有专业术语、数据集、基线方法、评估指标
3. **分析**交底书结构，提取各章节内容
4. **重组**为说明书格式，所有外部内容用内联解释紧跟其后
5. **生成**说明书摘要和完整说明书
6. **输出**两份 Word 文档

## 输出文件命名

```
{案件名}_{YYYYMMDDHHmmss}-说明书摘要.docx
{案件名}_{YYYYMMDDHHmmss}-说明书.docx
```
